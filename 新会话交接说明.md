# 新会话交接说明

把这份说明复制到新会话开头，或者把这个文件上传给新会话。它的目的不是让新会话“完全变成同一个实例”，因为不同会话无法继承内部记忆；它会尽量把当前协作背景、项目状态、审美方向、技术决策和接下来要做的动态博客目标讲清楚，让新会话接上现在的节奏。

```text
你现在接手的是我的 Firefly 博客项目。我之前已经和另一个 TRAE 会话一起改了很多内容，现在我希望你延续那个会话的协作方式和审美判断。

请先阅读当前文件夹里的《博客全面傻瓜式说明书.md》和《新会话交接说明.md》，再开始动手。不要急着重写项目。先理解当前静态博客的结构、已有改动、还原点和我想要的动态博客方向。

我喜欢的协作方式：
1. 先讲清楚你要做什么，再动代码。
2. 大改之前先创建还原点。
3. 每次改完运行 pnpm check。
4. 改动要保留现在的二次元、梦幻、液态玻璃风格。
5. 不要把页面改成普通后台模板。
6. 说明书只保留《博客全面傻瓜式说明书.md》这一份，新的说明也合并进去。

我现在想继续做的是：在新的文件夹里制作一个 Docker 部署的动态博客系统，最好能继承当前 Firefly 博客的视觉风格，并带后台管理文章、壁纸、相册、图片、音乐、模块配置等能力。
```

## 关于动态博客的目标

用户现在想做一个新项目，不是继续把当前静态博客无限改下去。

目标是：

```text
重新读取或选择一个新的文件夹
在新文件夹里制作一个 Docker 部署的动态博客
带后台管理
可以管理各种文件、图片、模块、文章、壁纸、相册、音乐等
尽量继承当前 Firefly 的视觉风格
```

如果新会话能选择新文件夹，建议创建一个干净项目，比如：

```text
DynamicBlog
```

或：

```text
FireflyDynamic
```

如果新会话只能继续当前文件夹，可以先在 Firefly 内创建原型子项目：

```text
Firefly > dynamic-blog
```

但更推荐新建独立文件夹，因为 Docker 动态博客会产生很多新文件。

## 动态博客推荐路线

不要一开始就重写成庞大的 CMS。推荐做：

```text
静态前台体验 + 动态后台管理 + Docker 部署
```

第一版可以这样理解：

```text
前台保持类似 Firefly 的博客体验和视觉风格
后台提供管理界面
后台可以上传图片、写文章、管理壁纸和相册
保存后数据进文件或数据库
Docker 负责运行前台、后台、数据库或文件服务
```

如果要更接近当前 Firefly，可以采用：

```text
前台：Astro 或 Next.js
后台：Next.js / SvelteKit / Vue + Node API
数据：SQLite 起步
文件：本地 uploads 目录
部署：Docker Compose
反代：Nginx 或 Caddy
```

更稳的第一版技术栈：

```text
Next.js 全栈
SQLite
Prisma
Docker Compose
本地 uploads 文件夹
管理员账号登录
```

原因：

```text
一个项目能同时做前台和后台
不用一开始拆太多服务
Docker 部署简单
SQLite 对个人博客够用
后期可以迁移 PostgreSQL
```

## 最重要的提醒

用户不是要一个普通模板网站。用户是在慢慢打磨一个有个人审美的博客。

要保持这些判断：

```text
能渐进改就不要推翻
能保留现有视觉就不要换模板
能写进总说明书就不要散落多个说明文件
能创建还原点就先创建还原点
能先讨论就不要直接做危险大改
```

动态博客可以大胆做，但要分阶段。第一版先让后台能登录、写文章、上传壁纸和相册、Docker 能跑起来。后面再逐步加模块管理、主题配置、音乐管理和发布系统。
